✅ SCHEDULER (background_scheduler.py)
────────────────────────────────────────
✓ APScheduler AsyncIOScheduler running 8 jobs
✓ All jobs wrapped in _safe_job_run() with async/sync bridge
✓ Redis-based distributed locking (fallback to in-memory)
✓ Job failure tracking with auto-pause after 3 consecutive failures
✓ 15-minute auto-resume after pause
✓ Misfire grace time set to 60 seconds to prevent job stacking
✓ Session cleanup with try/finally (not context manager) to prevent hanging

Jobs:
  1. confidence_monitor (every 30 min, 6 AM–11 PM IST)
     → Detects HIGH confidence disruptions
     → Initiates claims pipeline for affected workers
     → Publishes zone events to admin feed

  2. premium_recalculator (every Monday 00:00 IST)
     → Recalculates weekly premiums via XGBoost model
     → Updates policy records
     → Sends push notifications

  3. pool_health_auditor (every Sunday 23:59 IST)
     → Aggregates weekly payouts by zone
     → Updates pool balance snapshots
     → Triggers reinsurance alerts if utilization > 80%

  4. trust_score_updater (daily 02:00 IST)
     → Rewards clean workers (+0.02 trust)
     → Penalizes inactive workers (-0.01 trust)
     → Bounds trust score to 0.0–1.0 range

  5. stale_claim_resolver (every 5 minutes)
     → Auto-approves claims stuck in REVALIDATING > 2 hours
     → Auto-approves flagged claims with no strong fraud evidence
     → Triggers payouts and updates trust scores

  6. forecast_shield_job (every 6 hours + startup)
     → Fetches 48-hour weather forecasts from OpenWeatherMap
     → Stores risk windows in app.state.forecast_shields
     → Used by Forecast Shield feature to auto-upgrade coverage

  7. weather_alert_notifier (daily 07:10 IST)
     → Checks forecast shields for rain risk
     → Sends proactive weather alerts to workers

  8. disruption_scan (every 30 min, 6 AM–11 PM IST)
     → Calls check_disruptions_for_zone() for each zone
     → Fetches live weather (Open-Meteo) + AQI (OpenAQ)
     → Evaluates rain/heat/AQI thresholds
     → Queries admin-managed social events (curfew/strike)
     → Computes weighted confidence score
     → Only triggers claims pipeline for HIGH confidence
     → Zone-level isolation: one zone failure never stops others

Worker WebSocket Flow:
  1. Worker connects with JWT token
  2. Subscribes to personal channel: claim_updates:{worker_id}
  3. Receives real-time updates:
     - CLAIM_UPDATE (status changes)
     - DISRUPTION_ALERT (zone disruption detected)
     - PAYOUT_CREDITED (payout disbursed)
  4. Heartbeat every 30 seconds (PING/PONG)
  5. Auto-disconnect on 10-second PONG timeout

Admin WebSocket Flow:
  1. Admin connects with admin JWT token
  2. Subscribes to 4 channels:
     - all_claims_feed (all claim updates)
     - fraud_alerts (fraud ring detections)
     - zone_events (disruption detected/cleared, reinsurance triggers)
     - pool_health (pool balance updates)
  3. Receives multiplexed events with channel tags
  4. Real-time admin dashboard feed

✅ INTEGRATION FLOWS
────────────────────

FLOW 1: Onboarding → Policy → Payment → Activation
  1. Worker signs up via /api/v1/auth/register
  2. Creates policy via /api/v1/policies/create
  3. Pays premium via /api/v1/payments/initiate (Razorpay test mode)
  4. Policy status → "active"
  5. Worker appears in scheduler's eligible workers list

FLOW 2: Scheduler → Disruption → Claims → Payout → WebSocket
  1. disruption_scan job runs every 30 min
  2. Detects HIGH confidence disruption in zone
  3. Calls initiate_claims_for_disruption()
  4. For each eligible worker:
     a. Checks for duplicate claims (prevents double-payout)
     b. Calculates payout via PayoutEngine.compute_db_payout()
     c. Runs fraud check (4-layer engine)
     d. Decides: approved / review / rejected
     e. Creates Simulation + FraudFlag + FraudSignal rows
     f. Disburses payout (if approved)
     g. Updates worker trust score
     h. Publishes CLAIM_UPDATE via WebSocket
  5. Worker receives real-time notification on /ws/claims/{worker_id}
  6. Admin sees claim in /ws/admin/feed (all_claims_feed channel)

FLOW 3: Admin Event → Disruption → Claims Pipeline
  1. Admin creates social event (curfew/strike) via /api/v1/admin/events/create
  2. Event stored as DisruptionEvent with type "curfew" or "strike"
  3. disruption_scan detects it (is_active=True)
  4. Triggers claims pipeline same as FLOW 2
  5. Admin sees zone event in /ws/admin/feed (zone_events channel)

FLOW 4: Fraud Detection → Flagged Claims → Admin Visibility
  1. Claims engine runs check_fraud() (4-layer fraud engine)
  2. If fraud_score >= 0.3:
     a. Claim held for review (decision=REJECTED, not paid yet)
     b. FraudFlag rows created with flag_type + flag_detail
     c. FraudSignal row created with fraud_score
  3. stale_claim_resolver job (every 5 min):
     a. Checks claims stuck in REVALIDATING > 2 hours
     b. If no strong fraud evidence (score < FRAUD_THRESHOLD):
        - Auto-approves and triggers payout
        - Publishes CLAIM_UPDATE
  4. Admin sees fraud alert in /ws/admin/feed (fraud_alerts channel)
  5. Admin can review flagged claims via /api/v1/admin/fraud/queue
